# Internationalization & Localization (i18n/l10n) — Design Spec

**Date:** 2026-04-09
**Status:** Draft

---

## Context

Quorant's backend is currently English-only, USD-only, and UTC-only. All error messages, validation strings, and notification text are hardcoded in English. No locale preference exists on any model. No currency code is tracked on financial records.

This spec adds i18n/l10n infrastructure to make the application locale-capable. The initial deployment ships with a single `en_US` language pack. The architecture supports additional languages and currencies without further structural changes.

### Design Decisions

- **Codes-only API responses** — the server never returns translated human text in error responses. It returns structured message keys and interpolation params. Clients fetch a language pack from the server and translate locally.
- **Organization-level locale** — each org has a `locale` and `timezone`. Members inherit the org's locale. No per-user override in this iteration.
- **Server-hosted language packs** — the backend embeds language pack JSON files and serves them via a public API endpoint. Clients fetch the pack for their org's locale.

---

## Scope

### In Scope

| Area | Description |
|---|---|
| Error response format | Replace `message` with `message_key` + `params` in the API error envelope |
| Message key system | Structured keys for all 369 error/validation call sites across 20 patterns |
| Language pack API | `GET /api/v1/i18n/{locale}` serves embedded JSON language packs |
| en_US language pack | Complete English pack covering all message keys |
| Organization locale | `locale` and `timezone` columns on `organizations` table |
| Organization currency | `currency_code` column on `organizations` table |
| Financial currency codes | `currency_code` on all financial tables (assessments, ledger, payments, expenses, funds) |
| Notification template locale | `locale` column on `message_templates` for per-locale template resolution |
| Address country code | `country` column on `organizations` and `units` tables |

### Out of Scope

| Area | Reason |
|---|---|
| Per-user locale preference | HOAs are geographically bound; org-level is sufficient for now |
| Server-side message translation | Clients own rendering; server returns keys only |
| Right-to-left (RTL) support | No RTL locales planned in near term |
| Multi-currency accounting logic | Schema supports it, but business rules remain USD-only |
| Frontend translation infrastructure | Client-side i18n framework is a separate concern |

## 1. API Error Response Format

### Current Format

```json
{
  "errors": [{
    "code": "VALIDATION_ERROR",
    "field": "name",
    "message": "name is required"
  }]
}
```

### New Format

```json
{
  "errors": [{
    "code": "VALIDATION_ERROR",
    "message_key": "validation.required",
    "field": "name",
    "params": {"field": "name"}
  }]
}
```

### Error Struct Change

**File:** `backend/internal/platform/api/response.go`

```go
// Before
type Error struct {
    Code    string `json:"code"`
    Field   string `json:"field,omitempty"`
    Message string `json:"message"`
}

// After
type Error struct {
    Code       string         `json:"code"`
    MessageKey string         `json:"message_key"`
    Field      string         `json:"field,omitempty"`
    Params     map[string]any `json:"params,omitempty"`
}
```

### APIError Interface Extension

**File:** `backend/internal/platform/api/errors.go`

```go
// Before: each error type stores a Message string
// After: each error type stores a MessageKey + Params

type APIError interface {
    error
    StatusCode() int
    ErrorCode() string
    MsgKey() string
    MsgParams() map[string]any
}
```

### Error Constructor Changes

All error constructors change signature:

```go
// Before
func NewValidationError(message, field string) *ValidationError

// After
func NewValidationError(messageKey, field string, params ...Param) *ValidationError
```

Where `Param` is a simple key-value helper:

```go
type Param struct {
    Key   string
    Value any
}

func P(key string, value any) Param {
    return Param{Key: key, Value: value}
}
```

### WriteError Change

`WriteError` populates `MessageKey` and `Params` instead of `Message`:

```go
entry := Error{
    Code:       apiErr.ErrorCode(),
    MessageKey: apiErr.MsgKey(),
    Params:     apiErr.MsgParams(),
}
```

## 3. Language Pack API

### Package Structure

```
backend/internal/platform/i18n/
  i18n.go           // Pack type, loader, registry
  handler.go        // HTTP handler for GET /api/v1/i18n/{locale}
  locales/
    en_US.json      // Embedded language pack
```

### Pack Format

**File:** `backend/internal/platform/i18n/locales/en_US.json`

```json
{
  "locale": "en_US",
  "version": "1.0.0",
  "messages": {
    "validation.required": "{field} is required",
    "validation.invalid_uuid": "{field} must be a valid UUID",
    "validation.invalid": "invalid {field}",
    "validation.constraint": "{field} must be {constraint}",
    "validation.invalid_cursor": "invalid cursor",
    "validation.one_of": "{field} must be one of: {values}",
    "validation.at_least_one": "at least one field must be provided",
    "validation.invalid_json": "{field} is not valid JSON",
    "validation.invalid_body": "invalid request body: {detail}",
    "resource.not_found": "{resource} not found",
    "resource.conflict": "{resource} already exists",
    "auth.resolve_failed": "could not resolve user",
    "auth.missing_header": "missing {header} header",
    "auth.invalid_token": "invalid or expired token",
    "auth.invalid_header": "invalid authorization header format",
    "auth.missing_claims": "no claims in context",
    "access.insufficient_permissions": "insufficient permissions",
    "access.feature_unavailable": "feature {feature} is not available on your current plan",
    "access.no_organization": "no organization context",
    "rate_limit.exceeded": "too many requests",
    "unprocessable.not_implemented": "{feature} requires {dependency} (not yet implemented)",
    "unprocessable.request": "cannot process request",
    "server.internal_error": "an internal error occurred",
    "budget.invalid_status_transition": "budget must be in '{expected}' status to {action}, current status: {current}",
    "collection.case_not_open": "collection case is not in open status",
    "payment_plan.already_exists": "an active payment plan already exists for this case"
  }
}
```

### Embedding

```go
package i18n

import "embed"

//go:embed locales/*.json
var localeFS embed.FS
```

### Registry

```go
type Pack struct {
    Locale   string            `json:"locale"`
    Version  string            `json:"version"`
    Messages map[string]string `json:"messages"`
}

type Registry struct {
    packs map[string]*Pack // keyed by locale string
}

func NewRegistry() (*Registry, error) // loads all embedded packs
func (r *Registry) Get(locale string) (*Pack, bool)
func (r *Registry) Locales() []string // returns available locale codes
```

### HTTP Endpoint

```
GET /api/v1/i18n/{locale}
```

- **Unauthenticated** — language packs are public, needed before login
- **Response:** Full `Pack` JSON
- **404** for unsupported locale
- **Cache-Control:** `public, max-age=3600` (1 hour, packs change infrequently)
- **ETag** based on pack version for conditional requests

```
GET /api/v1/i18n
```

- Returns list of available locales: `{"locales": ["en_US"]}`

## 4. Organization Locale & Timezone

### Migration

```sql
ALTER TABLE organizations
  ADD COLUMN locale TEXT NOT NULL DEFAULT 'en_US',
  ADD COLUMN timezone TEXT NOT NULL DEFAULT 'UTC',
  ADD COLUMN currency_code TEXT NOT NULL DEFAULT 'USD',
  ADD COLUMN country TEXT NOT NULL DEFAULT 'US';
```

### Domain Changes

**File:** `backend/internal/org/domain.go` — Organization struct gains:

```go
Locale       string `json:"locale"`        // BCP-47 variant: en_US, es_MX
Timezone     string `json:"timezone"`      // IANA: America/New_York
CurrencyCode string `json:"currency_code"` // ISO 4217: USD, CAD
Country      string `json:"country"`       // ISO 3166-1 alpha-2: US, CA
```

### Request Changes

Create and update org requests gain optional `locale`, `timezone`, `currency_code`, and `country` fields with validation:
- `locale` validated against available packs in the i18n registry
- `timezone` validated via Go's `time.LoadLocation`
- `currency_code` validated as 3-character uppercase alpha
- `country` validated as 2-character uppercase alpha

## 6. Notification Template Locale

### Migration

```sql
ALTER TABLE message_templates
  ADD COLUMN locale TEXT NOT NULL DEFAULT 'en_US';

-- Update unique constraint to include locale
ALTER TABLE message_templates
  DROP CONSTRAINT IF EXISTS message_templates_org_id_template_key_channel_key,
  ADD CONSTRAINT message_templates_org_id_template_key_channel_locale_key
    UNIQUE (org_id, template_key, channel, locale);
```

### Resolution Order

When resolving a template for `(org_id, key, channel, locale)`:

1. Org-specific template for the requested locale
2. System default template for the requested locale
3. System default template for `en_US` (ultimate fallback)

### Domain Changes

`MessageTemplate` gains `Locale string json:"locale"`.

Template CRUD requests accept an optional `locale` field (defaults to `en_US`).

## 8. Call Site Migration Strategy

369 call sites need updating across 13 modules. The migration is mechanical:

### Pattern: `validation.required`

```go
// Before (130 sites)
api.NewValidationError("name is required", "name")

// After
api.NewValidationError("validation.required", "name", api.P("field", "name"))
```

### Pattern: `validation.one_of`

```go
// Before (7 sites)
api.NewValidationError(
    "frequency must be one of: monthly, quarterly, annually, semi_annually",
    "frequency")

// After
api.NewValidationError("validation.one_of", "frequency",
    api.P("field", "frequency"),
    api.P("values", "monthly, quarterly, annually, semi_annually"))
```

### Pattern: `resource.not_found`

```go
// Before (28 sites)
api.NewNotFoundError(fmt.Sprintf("organization %s not found", id))

// After
api.NewNotFoundError("resource.not_found",
    api.P("resource", "organization"),
    api.P("id", id.String()))
```

### Module Order

Migrate in dependency order (platform first, then leaf modules):

1. `platform/api` — error types, response envelope
2. `platform/middleware` — auth, rbac, ratelimit, entitlement
3. `platform/i18n` — new package (language packs, handler)
4. `iam` — user resolution errors
5. `org` — organization, unit, vendor, amenity, registration
6. `fin` — assessments, ledger, payments, budgets, expenses, funds, collections
7. `gov` — violations, hearings, ballots, meetings, ARB
8. `com` — announcements, threads, calendar, templates
9. `doc` — document management
10. `ai` — policy, compliance
11. `admin` — admin operations
12. `license` — licensing, billing
13. `webhook` — webhook management
14. `maint` / `task` / `audit` — remaining modules

## Verification

### Unit Tests

- `i18n.Registry` loads embedded packs, returns correct pack by locale, 404 for unknown
- `i18n` HTTP handler returns correct JSON, caching headers, 404 behavior
- All error types produce correct `message_key` and `params`
- `WriteError` serializes the new format correctly
- Org create/update validates locale, timezone, currency_code, country
- Template resolution follows locale fallback chain

### Integration Tests

- Create an org with `locale: "en_US"`, verify it persists and returns in GET
- Create financial records, verify `currency_code` defaults to org's currency
- Create templates with different locales, verify resolution order
- Fetch `GET /api/v1/i18n/en_US`, verify complete pack with all keys
- Fetch `GET /api/v1/i18n/xx_XX`, verify 404
- Trigger various validation errors, verify response contains `message_key` + `params` and no `message`

### Existing Test Updates

All existing handler and service tests that assert on error `message` strings must be updated to assert on `message_key` and `params` instead.
